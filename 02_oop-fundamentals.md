# 第2章：オブジェクト指向の基礎（なぜ設計図が必要か？）

Javaは「オブジェクト指向」という考え方で作られています。
極端に言えば、**「世の中のあらゆるものを『モノ（オブジェクト）』としてプログラム上で表現しよう」** という考え方です。
これを理解するために、まずは「クラス」という概念から見ていきましょう。

## この章でできるようになること

- クラスとインスタンスの違いを、実務でのデータ管理に結びつけて説明できる
- カプセル化が「不正な状態の防止」に効く理由を説明できる
- インターフェースを使う目的を、変更容易性と分業の観点で理解できる
- `static` とインスタンスメソッドの違いを説明できる

## 1. クラスとインスタンス（設計図と実体）

たい焼きを作る時、「たい焼きの型（金型）」に生地とあんこを入れて焼きますよね。

- **クラス（Class）＝ たい焼きの型（設計図）**  
  「たい焼きには、味（データ）という属性がある」「焼く（処理）ことができる」という定義だけを書いた設計図です。
- **インスタンス（Instance）＝ たい焼きそのもの（実体）**  
  設計図をもとに、実際にメモリ上に作り出された「クリーム味のたい焼き」や「あんこ味のたい焼き」です。

**【なぜ設計図を作るのか？】**  
顧客（ユーザー）を管理するシステムを作るとします。
設計図（クラス）を作っておけば、「ユーザーAさん」「ユーザーBさん」という実体（インスタンス）を無限に、しかも同じ規格で量産できるからです。

## 2. カプセル化（不正な操作を防ぐバリア）

クラスの中には、データ（変数の箱）と処理（メソッド）が入っています。
しかし、このデータを外から誰でも自由に変更できると、システムがおかしくなってしまいます。

例えば、銀行口座クラスで「残高」をマイナスに書き換えられては困ります。
そこで、**重要なデータを隠し、決まった手順（メソッド）からしか操作できないようにする**のが「カプセル化」です。

- `private`（プライベート）：自分自身のクラスからしか触れない（隠す）。
- `public`（パブリック）：どこからでも触れる（公開する）。

通常、**データ（変数）は `private` にして隠し、データを操作するための処理（メソッド）を `public` として公開**します。

## 3. インターフェース（「何ができるか」の約束事）

Spring Bootを使った開発で非常に多く登場するのが、この「インターフェース」です。
インターフェースとは、**「中に具体的な処理は書かず、『こういう操作ができますよ』という一覧（メニュー表）だけを定義した特別なクラス」** です。

例えば、「スマートフォン」のインターフェースには「電話をかける」「メールを送る」という名前だけが定義されています（iPhoneの中身なのかAndroidの中身なのかは関係ありません）。

**【なぜわざわざ中身のないメニュー表を作るのか？】**  
チーム開発で、「Aさんは処理を使う側」「Bさんは処理を作る側」だとします。
インターフェース（約束事）さえ決めておけば、AさんはBさんが処理の中身を完成させるのを待たずに、「こういう機能があるはずだ」と信じて自分のプログラムを書き進めることができるからです。
（※これは[第4章](04_di-and-annotations.md)の「DI（依存性の注入）」に深く関わってきます）

## 4. `static` とインスタンスメソッドの違い

第1章で `main` メソッドや `public static int add(...)` が登場しました。
ここでは、なぜ `static` が必要なのかを「クラス」と「インスタンス」の観点で整理します。

- **`static` メソッド**：クラスそのものに属するメソッドです。インスタンスを作らなくても呼び出せます。
- **インスタンスメソッド**：作成した実体（インスタンス）に属するメソッドです。`new` で実体を作ってから呼び出します。

`main` はプログラム開始時点で最初に呼ばれる必要がありますが、その時点ではまだ自分自身のインスタンスがありません。そのため `main` は `static` で定義されます。

```java
public class Sample {
    // クラスに属するメソッド（インスタンス不要）
    public static void sayStatic() {
        System.out.println("static method");
    }

    // インスタンスに属するメソッド（インスタンス必要）
    public void sayInstance() {
        System.out.println("instance method");
    }

    public static void main(String[] args) {
        // staticメソッドはクラス名から直接呼べる
        Sample.sayStatic();

        // インスタンスメソッドは new してから呼ぶ
        Sample sample = new Sample();
        sample.sayInstance();
    }
}
```

### 【サンプルコード】クラスとカプセル化

以下のコードは、「設計図（クラス）」を作り、カプセル化を使ってデータを守りながら「実体（インスタンス）」を操作する例です。

**ファイル名: `User.java`（設計図）**

```java
// ユーザーの設計図（クラス）
public class User {
    // データ（属性）は private にして外部から直接触れなくする（カプセル化）
    private String name;
    private int age;

    // 設計図から実体を作る時の初期設定ルール（コンストラクタと呼ぶ）
    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // データを見るための公開された処理（Getter）
    public String getName() {
        return this.name;
    }

    // 年齢を増やすための公開された処理
    public void addAge() {
        this.age = this.age + 1;
        System.out.println(this.name + "は" + this.age + "歳になりました。");
    }
}
```

**ファイル名: `Main.java`（設計図を使う側）**

```java
public class Main {
    public static void main(String[] args) {
        // 1. 設計図（Userクラス）から、「taro」という実体（インスタンス）を作る
        // 「new」というキーワードを使って実体を生み出す
        User taro = new User("太郎", 22);

        // 2. 公開された処理（publicメソッド）を使って操作する
        System.out.println("名前は " + taro.getName() + " です。");
        taro.addAge(); // 太郎が歳をとる処理を呼び出す

        // 3. 以下のコードはエラーになる（privateなデータに直接触ろうとしているため）
        // taro.age = -100; // <- 外から勝手に年齢をマイナスにされるのを防ぐ！
    }
}
```

### 【サンプルコード】インターフェース（メニュー表と中身）

Spring Bootでは、「呼び出す側」と「作る側」を切り離すためにインターフェースを多用します。

**ファイル名: `Calculator.java`（インターフェース＝メニュー表）**

```java
// 「calculate」という機能があることだけを約束し、中身は書かない
public interface Calculator {
    int calculate(int price);
}
```

**ファイル名: `TaxCalculator.java`（実装クラス＝実際の中身）**

```java
// メニュー表の約束（implements）を守って、具体的な計算処理を作る
public class TaxCalculator implements Calculator {
    @Override
    public int calculate(int price) {
        return (int)(price * 1.1); // 消費税10%の計算処理
    }
}
```

使う側（Controllerなど）は、「`TaxCalculator`」という具体的なクラス名を知らなくても、「`Calculator`が持っている`calculate`メソッド」を呼ぶだけでよくなります。これが第4章の「DI（依存性の注入）」の土台になります。
