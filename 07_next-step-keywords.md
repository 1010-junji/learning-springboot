## Java基礎の拡張

| 用語 | 簡易説明 |
|---|---|
| 継承（Inheritance） | 既存クラスの性質を引き継いで拡張する仕組み。共通化に有効だが、使い過ぎると構造が複雑になる。 |
| 抽象クラス（Abstract Class） | 共通処理を持ちつつ、一部実装を子クラスに任せるためのクラス。 |
| 例外処理（Exception Handling） | 異常系を `try-catch` で扱い、アプリの異常停止を防ぐ。 |
| コレクション（List / Map / Set） | 複数データを効率よく扱うための標準データ構造。 |
| ジェネリクス（Generics） | コレクション等の型安全性を高める仕組み。 |

## オブジェクト指向・設計

| 用語 | 簡易説明 |
|---|---|
| 移譲（Delegation） | 処理を別オブジェクトに任せる設計。継承の代替としてよく使う。 |
| ポリモーフィズム（多態性） | 同じインターフェースで実装を差し替えられる性質。DIの基盤にもなる。 |
| コンポジション | オブジェクトを組み合わせて機能を作る考え方。変更に強い設計を作りやすい。 |
| SOLID原則 | 保守しやすい設計のための5原則。特に単一責任と依存性逆転が重要。 |
| ドメインモデル | 業務の概念をコード上のモデルとして整理するアプローチ。 |

## Web / API / セキュリティ

| 用語 | 簡易説明 |
|---|---|
| REST | リソース指向でAPIを設計するスタイル。URLとHTTPメソッドの設計が重要。 |
| HTTPステータスコード | API結果を数字で表す規約（200, 400, 404, 500 など）。 |
| バリデーション | 入力値の妥当性をチェックする処理。`@Valid` などを使う。 |
| 認証 / 認可 | 「本人確認」と「権限確認」を分けて考えるセキュリティの基本。 |
| JWT | API認証でよく使うトークン方式。セッションレスな設計と相性が良い。 |

## Spring Boot実践

| 用語 | 簡易説明 |
|---|---|
| `@Configuration` / `@Bean` | Beanを明示定義するための仕組み。複雑な依存解決で使う。 |
| `@Profile` | 開発・検証・本番で設定やBeanを切り替えるための機能。 |
| `@Transactional` | 複数のDB処理を1つのまとまりとして成功/失敗させる仕組み。 |
| AOP（Aspect Oriented Programming） | ログ・監視・認可など横断的関心事を共通化する手法。 |
| Actuator | ヘルスチェックやメトリクス取得など運用監視機能を提供。 |

## データアクセス・テスト・運用

| 用語 | 簡易説明 |
|---|---|
| JPA / Hibernate | JavaオブジェクトとDBテーブルをマッピングするORM技術。 |
| N+1問題 | ORM利用時に発生しがちなクエリ過多問題。性能劣化の典型。 |
| 単体テスト（Unit Test） | クラス単位で振る舞いを検証するテスト。 |
| 結合テスト（Integration Test） | 複数コンポーネント連携を含めて検証するテスト。 |
| ログ集約 / 可観測性 | 本番障害を素早く追跡するためのログ・メトリクス・トレース設計。 |

## 学習の進め方（推奨）

- いま担当している機能に直接関係する用語から優先して学ぶ
- 1つの用語ごとに「定義」「どこで使うか」「使わないと何が困るか」をメモする
- サンプルコードを10〜20行で作って動かし、挙動を確認する
- 学んだことを第6章の質問テンプレート形式で言語化し、説明できる状態にする
